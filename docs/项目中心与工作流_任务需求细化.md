# 项目中心与工作流 · 任务需求细化

> 目标：以项目为中心，打通下载、构建、版本管理，形成完整可追溯的工作流，满足研发、测试、产品、运维等各类业务场景。

---

## 一、现状与问题

### 1.1 当前导航与数据流

| 模块 | 入口 | 数据来源 | 与项目的关系 |
|------|------|----------|--------------|
| 项目管理 | 管理中心 → 项目管理 | projects_db | 项目列表仅「任务」入口 |
| 项目详情 | 项目列表操作（部分入口） | project + tasks + project_versions_db | 仪表盘 + 任务 + 版本配置（非 APK 文件） |
| 任务 | /admin/projects/{id}/tasks | project_tasks_db | ✓ 按项目 |
| 版本管理 | 管理中心 → 版本管理 | APK 目录扫描 + changelog_db | 按项目分组，但为全局入口 |
| 构建管理 | 管理中心 → 构建管理 | Jenkins + 构建参数 | 跨项目，无默认项目上下文 |
| 下载中心 | 首页 /download-center | APK 目录扫描 + download_stats | 按项目分组，无管理中心入口 |

### 1.2 核心问题

1. **入口分散**：版本、构建、下载各自独立，进入项目后只能去「任务」，看不到版本/构建/下载的一体化视图。
2. **两套版本概念**：
   - **project_versions_db**：项目详情页的「版本管理」Tab，配置渠道、version_name、apk_path、jenkins_job 等，偏构建配置。
   - **APK 文件列表**：版本管理模块/下载中心，来自 `Config.APK_DIR` 扫描，按文件名解析 project/version。
3. **构建与项目脱节**：构建页需手动填 APP_NAME、VERSION_NAME 等，不支持「选定项目 + 版本配置 → 自动带参」。
4. **工作流断层**：构建完成 → APK 落盘 → 版本列表/下载中心能看到，但无「构建结果 → 版本 → 下载」的贯通路径。
5. **下载入口弱**：项目中心/任务页无直达「本项目可下载 APK」的快捷入口。

## 二、业务场景与角色诉求

### 2.1 研发 / 构建负责人

| 场景 | 诉求 | 当前满足度 |
|------|------|------------|
| 分支开发完成，要打测试包 | 选定项目+渠道 → 一键带参构建 | 部分：需手动填参 |
| 构建中 / 构建失败 | 在项目视角看到最近构建状态 | 否：需切到构建管理 |
| 构建成功，产物在哪 | 构建结果 → 直接看到对应 APK / 版本条目 | 部分：OUTPUT_BASE_DIR 可配，但无自动关联 |

### 2.2 测试 / QA

| 场景 | 诉求 | 当前满足度 |
|------|------|------------|
| 要测某个项目最新包 | 进项目 → 一眼看到可下载版本，扫码/下载 | 部分：需去下载中心按项目找 |
| 区分 dev / release 等渠道 | 按渠道筛选版本，知道哪个是推荐/线上 | 部分：版本管理有渠道，下载中心未显式区分 |
| 版本更新说明 | 下载前看 Changelog | ✓ 版本管理已支持 |

### 2.3 项目经理 / 产品

| 场景 | 诉求 | 当前满足度 |
|------|------|------------|
| 项目整体健康度 | 任务完成情况 + 最新版本 + 下载量概览 | 部分：项目详情有任务/版本，缺下载量 |
| 跨项目对比 | 哪个项目版本多、下载多 | 否：需切数据分析 |
| 发布节奏 | 最近构建/发布记录 | 部分：构建历史在构建页，非项目视角 |

### 2.4 运维 / 发布

| 场景 | 诉求 | 当前满足度 |
|------|------|------------|
| 标记推荐版本 | 某版本为当前推荐/线上 | ✓ changelog_db 有 recommended |
| 构建完成通知 | 飞书/站内通知 | ✓ Jenkins 实例可配 Webhook |
| 版本与构建关联 | 某版本对应哪次 Jenkins Build | 部分：project_versions 有 jenkins_job，未与 APK 文件名强关联 |

## 三、目标工作流（端到端）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. 需求/任务 → 2. 开发 → 3. 构建 → 4. 版本发布 → 5. 下载/分发 → 6. 数据分析   │
└─────────────────────────────────────────────────────────────────────────────┘
       任务管理           Jenkins        版本管理        下载中心        数据分析
         ↓                  ↓               ↓               ↓               ↓
    project_tasks    build_routes    project_versions   home.py      dashboard
                       + APK 目录      + APK 文件        (按项目)      (按项目)
```

**理想路径**：

1. **任务**：项目内创建任务，负责人开发。
2. **构建**：从项目中心或版本配置发起构建，参数由项目/版本配置带入。
3. **版本**：构建产物写入 APK 目录，自动或手动关联到项目版本；可编辑 Changelog、标记推荐。
4. **下载**：下载中心按项目展示，项目中心提供「本项目下载」快捷入口。
5. **数据**：下载行为记入 download_stats / download_events，数据分析按项目/版本统计。

## 四、任务需求细化（按优先级）

### 4.1 P0：项目中心一体化入口与导航

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P0-1 | **项目中心作为统一入口**：项目列表每行增加「进入项目中心」主入口，点击进入项目详情页（含仪表盘、任务、版本、构建、下载）。 | 项目列表默认主操作为「进入项目中心」，替代或 alongside「任务」。 |
| P0-2 | **项目详情页 Tab 结构**：仪表盘 | 任务 | 版本与发布 | 构建 | 下载，五个 Tab 清晰切换。 | 当前已有部分，补齐「构建」「下载」Tab，并统一风格。 |
| P0-3 | **「下载」Tab**：展示本项目下所有 APK（从 APK 目录扫描，filter by project_id），支持扫码、下载、Changelog 预览；与首页下载中心一致，但仅本项目。 | 用户无需离开项目即可下载本项目包。 |
| P0-4 | **「构建」Tab**：嵌入或跳转构建页，且 **URL 带 project_id**（如 `/admin/build?project=xxx`），构建页根据 project 自动填充 APP_NAME、OUTPUT_BASE_DIR 等（从 project 或 project_versions 取）。 | 构建参数默认与当前项目一致。 |

### 4.2 P0：项目列表增强

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P0-5 | **项目卡片/行信息**：展示任务数、APK 版本数、最近构建状态（可选）、下载量（可选）。 | 一眼判断项目活跃度。 |
| P0-6 | **快捷操作**：每行保留「任务」「版本」「构建」「下载」等快捷链接，可快速跳转对应 Tab 或模块。 | 高频操作一步可达。 |

### 4.3 P1：构建与版本的联动

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P1-1 | **构建参数与项目关联**：从项目详情进入构建时，从 project_versions 或 project 取 APP_NAME、OUTPUT_BASE_DIR、GIT_BRANCH 等默认值。 | 减少手工输入，降低出错率。 |
| P1-2 | **构建完成后的跳转**：构建成功页提供「查看本项目版本」或「下载」链接，回到项目版本/下载 Tab。 | 构建→版本→下载路径打通。 |
| P1-3 | **OUTPUT_BASE_DIR 规范**：构建输出目录与 APK 目录一致或为其子目录，产出文件名含 project 信息，以便 extract_project_name 正确解析。 | 构建产物能自动出现在版本列表与下载中心。 |
| P1-4 | **版本配置与 APK 的对应**：project_versions 中 apk_path 可指向具体文件或目录；版本管理展示时可区分「已配置版本」与「扫描到的 APK」。 | 便于理解哪些是「已发布」哪些是「临时构建」。 |

### 4.4 P1：下载体验强化

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P1-5 | **项目中心 → 下载中心**：项目详情页顶部或 Tab 内提供「在下载中心查看」链接，带 `?project=xxx` 筛选。 | 需要全局视图时快捷切换。 |
| P1-6 | **下载中心项目筛选**：URL 支持 `?project=xxx`，进入时默认只展示该项目。 | 便于分享、书签。 |
| P1-7 | **渠道/推荐标识**：下载列表项展示渠道（如 dev/release）、推荐角标（若 changelog 标记）。 | 测试/产品能快速识别版本类型。 |

### 4.5 P2：数据与统计在项目中心的体现

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P2-1 | **项目仪表盘数据**：在项目详情仪表盘增加「本项目下载量」「最近 7 日下载趋势」（可调用现有 analytics API 按项目筛选）。 | 项目经理在项目内即可看数据。 |
| P2-2 | **版本维度统计**：版本列表或下载 Tab 中，每个版本显示下载次数（来自 download_stats）。 | 了解哪个版本更受欢迎。 |
| P2-3 | **构建历史摘要**：项目构建 Tab 或仪表盘展示「最近 3 次构建」状态（成功/失败/进行中），点击可展开日志。 | 不必切到全局构建页。 |

### 4.6 P2：多渠道与版本标识

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P2-4 | **渠道筛选**：版本/下载列表中支持按渠道（dev/release 等）筛选，渠道可从 project_versions 或文件名约定解析。 | 满足多包体、多渠道场景。 |
| P2-5 | **推荐/当前版本**：列表中标出推荐版本、当前线上版本（若配置），置顶或角标突出。 | 明确「该下哪个」。 |
| P2-6 | **Changelog 与版本关联**：编辑 Changelog 时可选关联「文件名」或「版本配置 ID」，便于同一版本多包体共享说明。 | 避免重复编辑。 |

### 4.7 P3：扩展与优化

| 任务 ID | 需求描述 | 验收标准 |
|---------|----------|----------|
| P3-1 | **构建模板**：项目级或全局保存常用构建参数为模板，快速套用。 | 提升构建效率。 |
| P3-2 | **版本与 Git Tag 关联**：版本配置可填 Git Tag，构建时带入，便于追溯。 | 发布可追溯。 |
| P3-3 | **项目归档**：归档项目在列表中弱化展示，版本/下载可设为只读。 | 历史项目不干扰主流程。 |

## 六、数据与路由设计要点

### 6.1 路由结构（建议）

| 路由 | 说明 |
|------|------|
| `/admin/projects` | 项目列表 |
| `/admin/projects/<id>` | 项目详情（Tab：仪表盘/任务/版本/构建/下载） |
| `/admin/projects/<id>/tasks` | 任务列表（可保留独立页，或作为 Tab 内容） |
| `/admin/projects/<id>/versions` | 本项目版本（APK 扫描 + 版本配置） |
| `/admin/projects/<id>/build` | 本项目构建（或 redirect 到 /admin/build?project=id） |
| `/admin/projects/<id>/download` | 本项目下载（或 Tab 内嵌） |
| `/download-center` | 下载中心（支持 ?project=xxx） |
| `/admin/build` | 构建管理（支持 ?project=xxx 带参） |

### 6.2 关键数据流

- **APK 文件**：`Config.APK_DIR` 下的 `.apk`，文件名需含可解析的 project 信息（如 `{project}_{version}.apk` 或按 extract_project_name 规则）。
- **project_versions_db**：项目级版本配置，与 Jenkins 构建参数、apk_path 对应。
- **download_stats / load_download_events**：按 filename → project 聚合，支持按项目筛选。

---

## 七、非功能与约束

- **权限**：项目中心各 Tab 受 `can_view_project` / `can_edit_project` 控制；构建需 `build.trigger` scope。
- **性能**：APK 目录扫描可缓存，避免每次请求全盘扫描。
- **兼容**：现有 `/admin/versions`、`/admin/build`、`/download-center` 保留，项目中心作为「以项目为入口」的聚合视图，不破坏原有独立模块使用习惯。
