# 项目功能与商业化体检

## 当前功能面

### 公开侧
- 首页产品展示
- 产品详情页
- 登录后下载安装包
- Android / iOS 安装包区分展示与下载
- 二维码下载

### 管理后台
- 用户与权限管理
- 项目管理
- 版本管理
- 渠道与阶段管理
- Jenkins 实例管理
- 构建触发与日志查看
- 下载统计与数据分析
- 文档中心
- 审批、通知、报表、系统设置
- 产品内容管理
- 个人工作区

### 运维与交付
- Windows 云服务器一键部署
- 远程 Windows 部署
- HTTPS 反向代理脚本
- SQLite 同步、备份、恢复
- system check / smoke check

## 已完成的商业化基础

- Android / iOS 双平台安装包主链路已打通
- 统一权限服务 `services/authz.py`
- 安全头、CSRF、请求追踪、审计基础已接入
- 部分超大后台页已开始模板化
- 数据已具备 SQLite 同步与迁移能力
- 后台服务可稳定启动并在线验活

## 当前主要不足

### 功能缺失
- 缺少统一的“系统首页/运营驾驶舱”，还没有把异常、待审批、待发布、构建失败聚合到一个高价值入口
- 缺少对 Android / iOS 版本线差异的更细分管理，例如最低系统版本、商店包信息、分发方式说明
- 缺少真正的发布回滚界面与发布流水线可视化
- 缺少更完整的通知触达方式，例如邮件、Webhook、企业 IM

### 健壮度不足
- 核心业务仍是 JSON + SQLite 兼容模式，不是纯数据库主存储
- Jenkins 与启动相关逻辑里仍有较多 `except Exception: pass`
- 集成测试偏少，后台复杂流程覆盖不足
- 文档中心、版本工作流、部分后台大页仍有内联 HTML / JS

### 操作体验问题
- 后台模块很多，但缺少统一的信息架构和“推荐下一步”
- 某些页面成功/失败反馈仍偏技术化，不够面向运营人员
- 缺少批量操作后的汇总提示与错误明细
- 缺少更明确的平台筛选、版本筛选、负责人筛选的一致交互

### 视觉与商业标准差距
- 页面视觉语言不够统一，部分页面仍带历史样式
- 管理后台整体更像“工具集合”，还不像成熟商业产品
- 高价值页面缺少清晰的层级、状态色和重点信息突出

## 优先优化建议

### P0
- 数据库主存储化：逐步摆脱 JSON 主写入
- 收敛异常吞掉逻辑，重要失败必须可见
- 补关键后台流程集成测试
- 完成核心后台页模板化

### P1
- 建立统一后台导航、状态页和空态页
- 为 Android / iOS 版本管理补更细平台字段
- 建立发布流水线、审批流、回滚入口
- 补管理员常用操作的批量反馈与审计展示

### P2
- 做统一设计规范和后台视觉升级
- 接入外部通知与告警
- 建立多环境发布规范
- 建立恢复演练与值班手册

## 建议的下一批落地项

1. 继续模板化 `build_routes.py` 和 `docs_routes.py` 剩余大页
2. 为 `/admin` 做统一运营驾驶舱
3. 为 Android / iOS 版本补平台专属字段与校验
4. 为 Jenkins / 发布失败补更友好的错误提示
5. 扩展后台流程测试覆盖
