# 商业级改造路线图

## 目标

把当前下载中心从“可用”推进到“可长期运营、可多人协作、可上线交付”的状态，同时统一支持 Android 与 iOS 安装包。

## P0：两周内必须完成

1. 包件模型统一
- 所有扫描、上传、下载、统计统一按“安装包”处理。
- Android 使用 `.apk`，iOS 使用 `.ipa`。
- 页面、接口、后台列表都要显示平台标签。

2. 权限与安全收口
- 所有写操作统一走 `CSRF + 登录态 + 权限 + 审计日志`。
- 管理页高风险操作补二次确认与审计记录。
- 敏感配置全部通过 `.env` 或系统环境变量注入。

3. 启动与部署标准化
- Windows 云服务器一键部署脚本作为标准入口。
- Web 进程、后台任务、Jenkins 管理逻辑逐步拆分，避免启动阶段混入过多副作用。

4. 数据断链检查
- 产品和项目映射、版本和安装包映射、Jenkins 输出路径都增加体检检查。
- 部署后至少执行一次自检，避免页面空列表但实际上文件已存在。

## P1：一个月内完成

1. 数据层升级
- 核心业务数据从 JSON 文件迁移到 SQLite 或 PostgreSQL。
- 至少覆盖用户、项目、版本、下载记录、审批、通知。

2. 后台页面拆分
- 把超长 `render_template_string` 页面拆成模板文件和静态 JS。
- 平台筛选、版本编辑、下载页等前端交互统一组件化。

3. 工作流完善
- 版本记录增加平台字段，区分 Android / iOS 版本线。
- Jenkins 构建参数、产物路径、发布记录与平台绑定。
- 下载统计支持按平台查看趋势。

4. 测试补齐
- 增加 Android / iOS 双平台测试样例。
- 覆盖上传、下载、产品页展示、API 统计、权限控制。

## P2：三个月内完成

1. 观测与告警
- 结构化日志、请求 ID、后台任务状态、慢请求监控。
- 构建失败、下载异常、登录异常增加告警通知。

2. 多环境与备份
- 明确开发、测试、生产配置边界。
- 建立数据备份、安装包备份、恢复演练流程。

3. 商业交付能力
- 完善部署、升级、回滚、故障处理手册。
- 补操作审计导出、版本追溯、安装包留档策略。

## 验收标准

满足以下条件才算达到本轮目标：

1. 首页、产品页、项目页、版本页都能清楚区分 Android 和 iOS。
2. `.apk` 与 `.ipa` 都能上传、展示、下载、计数。
3. 匿名访问不会泄露受限安装包列表。
4. Windows 云服务器可一键部署并可用。
5. 核心测试通过，且至少完成一次真实启动验证。
