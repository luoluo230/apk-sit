# 星云游戏站完整版产品策划案

## 1. 项目目标

本项目不是单一官网，而是一套面向商业化发行与长线运营的双端平台：

- 对外：品牌官网、公司简介、游戏详情、新闻公告、福利中心、玩家论坛、下载中心
- 对内：管理工作台、产品中心、项目中心、审批中心、官网配置中心、角色化后台

本策划案用于约束后续落地，任何页面或功能只有在“已开发 + 已验证”后，才允许计入完成状态。

## 2. 设计原则

### 2.1 前台原则

- 前台必须具备品牌感，不能是后台模板换皮
- 玩家端页面不得暴露开发者登录入口、后台路径、内部配置提示
- 首页必须承担品牌、转化、内容入口、媒体展示四类职责
- 新闻、福利、论坛必须是完整内容中心，而不是零散模块

### 2.2 后台原则

- 后台首页按角色和职责分区，不再大杂烩
- 运维、运营、开发、配置必须有明确默认入口
- 发布类内容必须先审批后上线
- 删改操作要支持归档、回退、审计

### 2.3 项目隔离原则

- 新闻、福利、官方帖子必须绑定产品和项目
- 不同项目内容不能互相污染
- 玩家论坛允许存在全站讨论区，但官方内容必须按项目隔离

### 2.4 验收原则

- 每个页面必须能真实访问
- 每个功能必须至少走通一条真实链路
- 只改文档、不改运行页，不计入完成
- 只改样式、不改功能，不计入完成

## 3. 角色与权限规划

### 3.1 玩家

- 浏览首页
- 浏览公司简介
- 浏览游戏详情
- 浏览新闻、福利、论坛
- 发帖与评论

### 3.2 运营

- 创建新闻
- 创建福利
- 创建官方帖子
- 编辑草稿
- 提交审批
- 查看自己提交的记录

### 3.3 审批人

- 查看待审批内容
- 审批通过
- 驳回并填写意见
- 查看审批历史

### 3.4 开发

- 维护产品资料
- 维护下载入口与版本资源
- 管理项目、版本、媒体资源

### 3.5 运维

- 管理 Jenkins
- 查看构建记录
- 查看服务健康、日志、报表

### 3.6 管理员

- 管理用户与角色
- 管理官网配置
- 管理公司资料
- 管理系统设置与审计日志

## 4. 对外信息架构

### 4.1 首页 `/`

目标：

- 建立品牌印象
- 承接主推游戏
- 提供新闻、福利、论坛、下载入口

页面结构：

- Hero 主视觉区
- 主推游戏区
- 公司实力区
- 最新新闻区
- 福利预览区
- 社区热帖区
- 媒体展示区

验收标准：

- 顶部导航包含品牌、新闻、福利、论坛、下载中心
- 首页首屏能直接进入游戏详情和下载中心
- 首页展示最近新闻和福利入口
- 首页不出现开发者登录暴露文案

### 4.2 公司简介 `/about/company`

目标：

- 展示品牌故事与企业背书

页面结构：

- 品牌介绍
- 发展历程时间线
- 关键成就
- 团队理念
- 联系方式

验收标准：

- 页面独立存在，不依赖首页锚点
- 内容可由后台配置修改

### 4.3 游戏详情 `/product/<id>`

目标：

- 展示单个游戏项目的完整对外信息

页面结构：

- 游戏介绍
- 媒体展示
- 下载入口
- 商店入口
- 平台信息
- 最新新闻
- 当前福利
- 官方社区入口
- 玩家论坛入口

约束：

- 不允许出现开发者登录入口
- 不允许出现后台配置提示

### 4.4 新闻中心 `/news` 与 `/news/<id>`

目标：

- 建立标准新闻与公告中心

能力要求：

- 首页展示最近 5 条
- 列表页分页
- 按项目筛选
- 按类型筛选
- 详情页展示标题、时间、正文、项目归属

### 4.5 福利中心 `/welfare` 与 `/welfare/<id>`

目标：

- 为玩家提供独立福利领取入口

能力要求：

- 列表页展示礼包码、有效期、状态、项目归属
- 支持项目筛选
- 详情页展示完整说明

### 4.6 玩家论坛 `/forum` 与 `/forum/<id>`

目标：

- 建立官方内容和玩家内容共存的社区

能力要求：

- 官方帖子与玩家帖子共存
- 支持发帖、评论
- 支持项目分区
- 支持分类、置顶、隐藏
- 支持游客或普通玩家发帖

## 5. 对内后台信息架构

### 5.1 管理首页 `/admin`

目标：

- 统一工作台，不再让用户在多个后台页来回切

页面结构：

- 第一屏展示待办、审批、风险、关键统计
- 第二屏展示模块入口
- 右侧展示安装包概览、风险提醒、审计记录

### 5.2 运维中心

- Jenkins
- 构建任务
- 版本分发
- 下载服务
- 健康监控
- 日志与报表

### 5.3 运营中心

- 新闻管理
- 福利管理
- 官方帖子管理
- 社区审核
- 审批中心
- 通知中心

### 5.4 开发中心

- 项目管理
- 产品管理
- 版本资源
- 媒体资源

### 5.5 配置中心

- 官网配置
- 公司资料
- 外部模块入口
- 登录页配置

## 6. 玩家生态管理规划 `/admin/community`

目标：

- 把新闻、福利、官方帖子、玩家治理统一放在一个项目隔离的内容台

要求：

- 顶部可按产品筛选
- 新闻、福利、官方帖子三类表单支持新建和编辑
- 创建后必须提交审批
- 列表支持编辑、删除、置顶、隐藏、恢复
- 页面中不再放“玩家官网配置”和“开发者官网配置”的编辑入口，只提供跳转到配置中心

## 7. 产品中心规划 `/admin/products`

目标：

- 产品中心负责“对外展示主数据”，不是只维护一个卡片

产品中心职责：

- 产品基础信息
- 展示文案
- Hero 素材
- 平台支持
- 下载入口
- 商店链接
- 关联新闻
- 关联福利
- 关联官方帖子
- 关联媒体资源

关系约束：

- 产品面向官网展示
- 项目面向研发与版本管理
- 内容必须挂靠产品

## 8. 审批流规划

适用对象：

- 新闻
- 福利
- 官方帖子
- 重要官网配置变更

状态机：

- 草稿
- 待审批
- 已发布
- 已驳回
- 已归档

流程：

1. 运营创建内容
2. 提交审批
3. 审批人查看并处理
4. 通过后发布，驳回后回到编辑
5. 已发布内容再次编辑时重新走审批

## 10. 分阶段实施清单

### Phase 1：基础结构修复

- 清理前台暴露的开发者入口
- 清理乱码、坏文案、旧配置残留
- 修复后台首页第一屏
- 修复玩家生态管理页

### Phase 2：内容中心完成

- 新闻中心
- 福利中心
- 玩家论坛
- 公司简介
- 游戏详情

### Phase 3：审批与项目隔离

- 新闻审批链
- 福利审批链
- 官方帖子审批链
- 项目筛选与内容隔离

### Phase 4：后台产品化

- 产品中心重构
- 官网配置中心
- 角色化入口
- 审计与风险提示

## 11. 最终验收标准

- 首页、公司页、新闻页、福利页、论坛页、产品页全部可访问
- 管理首页、玩家生态页、审批中心、产品中心、配置中心全部可访问
- 新闻、福利、官方帖子都能从后台提交审批并在通过后进入前台
- 玩家论坛可以发帖和评论
- 页面不再出现明显乱码、`????`、旧后台文案残留
- 玩家页面不再暴露开发者后台入口
