# APK 下载中心 · 管理中心商业级产品策划案

本文档对管理中心各模块做**商业级**产品策划，面向内部交付、对外商用或 SaaS 化时的需求对齐与迭代依据。  
**仅做需求与设计说明，不改变现有代码逻辑；实施时以增量为原则。**

**现有功能与交互的整合说明**：各章均包含「现有功能清单与交互要点（已实现）」或「现有功能与交互全景」，对当前已上线的功能、界面元素、操作路径、视觉与交互细节做完整记录，**保证在向商业级演进时不丢失任何已有能力与体验**；商业级目标以「核心功能清单」「扩展方向」等形式单独列出，便于对照实施。

### 0.2 层级关系与导航结构

管理中心采用**一级分组 + 二级模块**的层级，便于菜单折叠与权限分组配置：

```
管理中心
├── 人员与权限
│   └── 用户管理
├── 项目与协作
│   ├── 项目管理
│   ├── 我的任务          ← 可为顶部独立入口，同时在分组内保留
│   ├── 通知中心          【新增】
│   └── 审批管理          【新增】
├── 发布与资产
│   ├── 构建管理
│   ├── 版本管理
│   └── Jenkins 管理
├── 数据与洞察
│   ├── 数据分析
│   └── 报表中心          【新增】
└── 治理与运维
    ├── 操作日志
    └── 系统设置          【新增】
```

### 0.3 分组说明与模块依赖

| 分组         | 定位                     | 典型依赖关系                         |
|--------------|--------------------------|--------------------------------------|
| 人员与权限   | 账号与权限底座           | 被所有分组依赖                       |
| 项目与协作   | 项目维度与个人协作       | 依赖用户、项目；任务/审批依赖项目    |
| 发布与资产   | 构建、版本、Jenkins      | 依赖项目、用户；版本依赖 APK 目录   |
| 数据与洞察   | 统计、报表、导出         | 依赖下载事件、项目、版本             |
| 治理与运维   | 审计、配置、安全策略     | 操作日志被各模块写入；设置影响全局  |

- **我的任务**：既可作为「管理中心 → 项目与协作」下的一项，也可在顶部导航或首页提供独立入口，便于高频访问。
- **通知中心**：各模块产生的提醒、@、审批结果等汇总展示，依赖用户、任务、审批等。
- **审批管理**：可驱动版本发布、敏感操作（如删除项目）的二次确认或多人审批，依赖项目、用户。
- **报表中心**：定时报表、报表模板、导出记录，与「数据分析」互补（数据分析偏即时查询，报表中心偏定期推送与沉淀）。
- **系统设置**：仅超级管理员或管理员可访问，配置登录限流、日志保留、通知渠道、功能开关等，影响全平台。

### 0.4 整体架构、安全与多平台/多项目适配

**架构原则**  
- **前后端**：服务端渲染（SSR）为主，关键操作走 JSON API，保证无 JS 时仍可浏览与基础操作；前端统一 Tailwind + Font Awesome，风格与管理中心一致。  
- **数据与权限**：项目维度隔离（viewers/editors）；首页仅展示当前用户有权限查看的项目下的 APK；操作日志覆盖关键写操作，便于审计与排障。  
- **配置与部署**：核心配置集中到 `.env`，支持 `PUBLIC_URL`/`EXTERNAL_DOMAIN` 外网访问；部署脚本（deploy.sh / deploy.bat）一键完成目录与配置检查，便于在 Windows / macOS 等不同平台复现。

**安全要点**  
- 登录限流与锁定、Session Cookie（HttpOnly / SameSite / 外网 HTTPS 时 Secure）、安全响应头（X-Frame-Options、X-Content-Type-Options 等）、密码仅存哈希、敏感操作写审计日志。  
- 上传与下载路径校验、项目可见性校验（can_view_project）在下载与首页展示时统一使用，避免越权。

**多平台与多项目适配**  
- **多平台**：部署与运行不依赖特定 OS；文档与脚本区分 Mac/Linux 与 Windows；健康检查 `/health` 供负载均衡与监控。  
- **多项目**：项目 ID 作为 APK 分组与权限边界；不同项目可配置不同参与人；首页与管理中心按「有权限的项目」过滤，保证同一套代码适配单项目与多项目场景。

### 0.5 主界面与首页设计规范

**设计目标**  
- **美观**：统一配色（如 slate/indigo 与白底）、圆角与阴影、清晰层级；首屏信息密度适中，不拥挤。  
- **操作友好**：顶部导航固定，入口明确（首页、我的任务、管理中心、用户与退出）；搜索醒目；按项目分组展示 APK，下载与扫码操作一步可达；弹窗（扫码、上传）风格与整站一致，关闭方式明确。

**首页结构（已实现与建议）**  
- **顶部导航**：左侧产品名/Logo，右侧当前用户名、「我的任务」链接（有项目管理权限时）、「管理中心」链接（有权限时）、「退出」。  
- **汇总区**：总 APK 数、总下载量、总大小、活跃项目数；卡片式、图标+数字+标签，响应式多列。  
- **搜索与操作栏**：搜索框（按 APK 名称/项目实时过滤）、刷新；管理员显示「上传 APK」按钮。  
- **内容区**：按项目分组；每组展示项目名称（可选项目图标）；组内为 APK 列表，每项展示名称、版本、大小、日期，操作为「扫码」「下载」。  
- **弹窗**：扫码弹窗展示二维码；上传弹窗为表单+进度，成功后刷新列表。  
- **权限与数据**：仅展示当前用户有权限查看的项目下的 APK；统计数字与列表数据一致，保证逻辑完善、安全。

**当前实现要点**  
- 首页已按 **can_view_project** 过滤，仅展示有权限项目下的 APK；总 APK 数、总下载量、总大小、活跃项目数均基于可见数据计算。  
- 顶部导航：产品名 + Logo 区；右侧为当前用户、「我的任务」（有项目管理权限时）、「管理中心」（管理员）、「退出」；风格与 slate/indigo 配色统一。  
- 汇总区为四张统计卡片（圆角、图标、hover 微动）；搜索栏支持按 APK 名称或项目实时过滤，并联动隐藏无匹配的项目组。  
- 按项目分组展示：每组为卡片（项目图标/占位、项目名、包数量），组内为 APK 列表（名称、版本、大小、日期 + 扫码/下载）；无可见 APK 时展示友好提示。  
- 扫码与上传弹窗：圆角、阴影、遮罩；上传支持进度与成功刷新。

## 二、用户管理 · 商业级策划

### 2.1 模块目标

- 建立**统一账号与角色体系**，支持超级管理员、管理员、普通用户。
- 支持**按模块授权**（非管理员可仅开通「项目管理」「我的任务」等），最小权限、职责清晰。
- 满足**安全与合规**：密码不可逆存储、登录限流、禁用账号、操作可审计。

### 2.2 目标用户

- **超级管理员 / 管理员**：配置用户、分配模块权限、重置密码、启用/禁用账号。
- **普通用户**：仅使用被授权的模块，无用户管理入口。

### 2.3 核心功能清单（商业级）

| 功能点           | 说明                                     | 优先级 |
|------------------|------------------------------------------|--------|
| 用户列表         | 分页/搜索（用户名、角色）、展示最后登录  | P0     |
| 新增用户         | 用户名唯一校验、初始密码、角色与模块勾选 | P0     |
| 编辑用户         | 修改角色、模块权限、禁用/启用            | P0     |
| 重置密码         | 仅管理员操作，记录审计                   | P0     |
| 删除用户         | 保护 super_admin/admin，禁止删除当前登录 | P0     |
| 模块权限配置     | 勾选式多选（项目管理、构建、数据分析等）  | P0     |
| 「全部开通」     | 除「用户管理」外一键全选                 | P1     |
| 批量导入/导出    | CSV 导入账号、导出列表（脱敏）           | P2     |
| 密码策略         | 最小长度、复杂度、定期提醒修改           | P2     |
| 双因素认证(2FA)  | TOTP 可选绑定                             | P3     |

### 2.4 数据模型要点

- **用户**：username（唯一）、password（哈希）、role（super_admin | admin | user）、allowed_modules（列表或 *）、disabled、created_at、last_login、email（可选）。
- **约束**：user_management 仅 super_admin/admin 可访问；普通用户可见模块由 allowed_modules 决定。

### 2.5 权限与安全

- 仅 **super_admin / admin** 可进入用户管理；所有增删改、重置密码写入**操作日志**。
- 密码存储：仅保存哈希（如 SHA-256 或更强），禁止明文或可逆加密。
- 登录失败限流、锁定时长可配置（已有）；可扩展：同一账号多端登录策略、会话超时。

### 2.6 交互与体验

- 列表支持按用户名/角色筛选、排序；禁用用户视觉区分（如灰色或标签）。
- 编辑/新增弹窗内模块以多选框展示，并注明「用户管理仅管理员可见」。
- 重置密码后提示「已重置，建议用户首次登录修改」。

### 2.7 扩展方向

- 与 LDAP/SSO 对接（只读同步或登录代理）。
- 角色模板：预设「项目经理」「测试负责人」等，一键套用模块权限。
- 操作日志中增加「操作对象」字段，便于按用户维度检索。

### 2.8 现有功能清单与交互要点（已实现）

**功能清单**  
- 用户列表：表格展示用户名、角色、模块权限、状态、最后登录、创建时间、操作；支持实时搜索（用户名、角色）过滤，无分页，前端过滤。  
- 添加用户：顶部「+ 添加用户」展开表单；用户名、密码、角色（管理员/普通用户）；角色为普通用户时显示「除用户管理外全部」复选框及按模块多选（项目管理、构建管理、数据分析、版本管理、Jenkins 管理、操作日志）；提交后校验用户名非空、密码非空、用户不存在，写入操作日志。  
- 编辑用户权限：弹窗「编辑用户权限」，展示用户名；除用户管理外全部 + 各模块多选；取消某模块勾选时自动取消「全部」；「禁用该账号」复选框；保存后写操作日志。  
- 重置密码：弹窗，新密码输入框 +「随机生成并复制到剪贴板」；提示「密码已加密存储，无法查看」；确定重置后写操作日志。  
- 删除用户：二次确认；禁止删除 admin、禁止删除超级管理员；删除后写操作日志。  
- 启用/禁用：列表操作「启用」「禁用」按钮，仅对非 super_admin 且非 admin 显示；更新后写操作日志。

**交互与视觉**  
- 列表：表头灰底、斑马行（白/浅灰）、行 hover 浅蓝；状态用徽章（已禁用红底、正常绿底）；操作按钮为文字链接样式（编辑权限蓝、重置密码紫、禁用琥珀、删除红），多按钮 flex 换行。  
- 弹窗：居中、圆角、阴影，点击遮罩关闭；编辑权限弹窗内模块为两列网格多选，说明文案「取消勾选某一项时将自动取消全部」。  
- 表单：输入框 focus 环蓝色；添加用户时角色切换显示/隐藏模块权限区域。

## 三、项目管理 · 商业级策划

### 3.1 模块目标

- 支持**多项目**独立配置，每个项目具备名称、简介、阶段、图标、Git/网络等元数据。
- 明确**查看者与编辑者**，编辑者可配置参与人及项目内角色（如策划、开发、测试），为任务分配与权限提供依据。
- 与「任务」「版本」「数据分析」打通：任务按项目归属、版本与统计按项目筛选。

### 3.2 目标用户

- **管理员 / 被授权用户**：创建/编辑/删除项目，维护参与人及角色。
- **项目参与人**：在项目内创建/处理任务、查看项目下版本与统计。

### 3.3 核心功能清单（商业级）

| 功能点           | 说明                                           | 优先级 |
|------------------|------------------------------------------------|--------|
| 项目列表         | 卡片或列表、搜索、按阶段/状态筛选              | P0     |
| 创建项目         | 项目 ID、名称、阶段、参与人及角色              | P0     |
| 编辑项目         | 修改元数据、参与人增删改、角色调整             | P0     |
| 删除项目         | 二次确认，可选级联提示（存在任务时的处理策略）| P0     |
| 参与人管理       | 添加/移除/改角色，用户名存在性校验             | P0     |
| 项目内角色       | 固定列表（如策划、开发、测试、其他）或自定义  | P0     |
| 查看者/编辑者    | 仅查看者只能看项目与任务，不能改配置与任务     | P1     |
| 项目归档         | 归档后只读，不参与「我的任务」默认列表         | P2     |
| 项目模板         | 从模板创建项目，预填阶段与角色                 | P2     |

### 3.4 数据模型要点

- **项目**：id、name、name_en、phase、icon、intro、detail、viewers、editors、member_roles（user -> role）、network_connection、git_url、git_ssh_key_path、created_at、status（active | archived）等。
- **与任务**：任务归属 project_id；任务「当前负责人」可来自 member_roles 中的用户。

### 3.5 权限与安全

- 列表与详情：用户需在 **viewers 或 editors** 中，或为管理员。
- 创建/编辑/删除：仅 **editors 或管理员**；删除项目记操作日志，并考虑任务数据的保留或清理策略。

### 3.6 交互与体验

- 参与人添加时实时校验用户名是否存在、是否已禁用，避免无效提交。
- 角色选择用下拉或单选，与任务创建/流转时的「负责人」选项一致。
- 项目卡片展示成员数、任务数（若已有关联统计），便于快速判断项目活跃度。

### 3.7 扩展方向

- 项目级配置：是否开启任务、是否开启版本发布审批等。
- 项目分组/标签，用于列表筛选与报表。
- 与外部系统（Jira、GitLab）的项目 ID 映射，便于后续打通。

### 3.8 现有功能清单与交互要点（已实现）

**功能清单**

### 7.5 权限与安全

- 用户仅能看见自己为**当前负责人**的任务；编辑/流转/评论需满足项目内编辑或任务创建者/负责人规则。

### 7.7 扩展方向

### 7.8 现有功能清单与交互要点（已实现）

**功能清单**  
- 任务范围：当前用户为「当前负责人」的任务，且状态不为「已完成」；已作废任务会显示（仅过滤 done）。

- 排序：首要（剩余时间、状态、紧急程度、开始时间、负责人）通过链接切换；次要通过下拉（无/剩余时间/状态/紧急程度/开始时间/负责人）onchange 提交；稳定排序含任务 ID 作为第三键。

- 行样式：按状态底色（已作废灰、尚未开始 slate、进行中蓝/红/琥珀/绿依进度、待验收紫、验收通过绿、验收未通过橙）；超时/严重超时左侧红边、即将超时琥珀边。

### 现有功能清单与交互要点（已实现）

- 构建参数区：使用 Jenkins 实例（下拉，来自 `/api/jenkins-manage/list`，选项为「默认」+ 各实例「端口 任务名 - 运行中/已停止」）；APP_NAME、VERSION_NAME、VERSION_CODE、UNITY_VERSION（可为下拉，由实例 build_defaults.unity_versions 填充）、OUTPUT_BASE_DIR、GIT_BRANCH（可为下拉，由实例 build_defaults.git_branches 填充）；开始构建、停止构建（构建中时显示）。

- 构建日志：下方大块 pre，绿底黑字；点击历史或触发后自动刷新；轮询间隔 3 秒直至该次构建结束。

**交互与视觉**  
- 两栏布局（参数+历史 | 下方整行日志）；按钮「开始构建」「停止构建」显隐切换；状态文案绿色/红色；历史列表为链接+徽章。

### 11.5 权限与安全

- 仅具备「报表中心」模块权限的用户可访问；模板可见范围受项目权限约束（只能选有权限的项目）。
- 导出与推送行为记操作日志；大批量或高频导出可限流。

### 11.6 交互与体验

- 模板列表支持搜索与筛选；创建/编辑模板时用与数据分析一致的筛选与图表配置界面。
- 定时报表配置界面简洁：选模板、选频率、选接收人；导出记录支持按用户、时间筛选。

### 11.7 扩展方向

- 报表内容支持「自定义维度」或与埋点数据结合。
- 推送至飞书/企业微信，生成摘要卡片。
- 报表版本留存：每次定时生成保留一份快照，便于历史对比。

---

## 十二、非功能要求（商业级共性）

### 12.1 安全

- 所有敏感操作（用户/项目/任务增删改、密码重置、构建触发等）写入操作日志。
- 会话超时、Cookie 安全（HttpOnly、SameSite、HTTPS 下 Secure）按现有安全加固执行。
- 接口与页面按权限校验，避免越权访问。

### 12.2 性能与可用性

- 列表分页或虚拟滚动，避免单页数据过大。
- 导出与复杂统计可异步化，避免阻塞请求；大表查询可加索引或预聚合。
- 关键配置与数据有备份或持久化策略（如 data 目录定期备份）。

### 12.3 可维护与可扩展

- 模块与路由清晰，新增操作类型时在操作日志中统一扩展 action 枚举与展示文案。
- 配置项（如登录限流、日志保留天数）通过环境变量或管理后台可配，避免写死。

### 12.4 兼容与迁移

- 数据文件（users、projects、tasks、audit_log 等）结构变更时提供迁移脚本或兼容读取，保证从旧版本升级可平滑过渡。
- 新机器部署遵循《DEPLOY.md》，最少命令完成安装与配置。

## 十四、文档与迭代

- 本文档作为**产品需求与设计基准**，与《PRODUCTION_CHECKLIST.md》《DEPLOY.md》并列：  
  - 策划案侧重「做什么、为什么、做到什么程度」；  
  - 清单侧重「做到没有、验收标准」；  
  - 部署文档侧重「怎么装、怎么迁、怎么对外」。
- 后续若新增模块，可在此文档中新增章节，并保持与现有模块的依赖与权限关系一致；当前已包含通知中心、审批管理、系统设置、报表中心四类新增模块的策划。
- 所有功能迭代以**增量、不删减已有能力**为原则，避免破坏现有使用习惯与集成方。

## 十五、商业级模块缺口分析

在现有 12 个模块基础上，从**组织与人员、协作与交付、数据与洞察、治理与运维、体验与开放**五个维度做缺口分析，列出商业级产品中常见、当前策划尚未单独成模块或未覆盖的能力，并给出与现有模块的关系及建议优先级（P0 强烈建议补、P1 建议补、P2 可选）。

---

### 15.1 组织与人员维度

| 缺口模块 | 商业级为何常见 | 与现有模块关系 | 建议优先级 |
|----------|----------------|----------------|------------|
| **组织架构 / 部门管理** | 多部门、多团队企业需要「部门→用户」「部门→项目」的归属与统计；权限可按部门继承；报表常按部门维度。 | 用户管理当前是「平铺用户+角色+模块」；项目有参与人但无「项目所属部门」。组织架构可成为用户与项目的上层维度。 | **P1** |
| **角色/权限模板** | 快速给新用户套用「项目经理」「测试负责人」等预设，减少逐项勾选；权限变更时批量应用模板。 | 用户管理已有「全部开通」和按模块勾选，但无命名模板与复用。可作用户管理的增强子能力，或独立「角色模板」配置页。 | **P2** |
| **个人中心 / 个人设置** | 修改密码、头像、通知偏好、界面语言/主题等应在「个人」入口完成，与「系统设置」（全局、仅管理员）区分清晰。 | 当前修改密码可能在用户管理（管理员重置）或分散在登录/安全流程；通知偏好、头像等若没有则缺统一入口。独立「个人中心」可集中：资料、密码、通知偏好、操作记录（仅本人）。 | **P0** |

---

### 15.2 协作与交付维度

| 缺口模块 | 商业级为何常见 | 与现有模块关系 | 建议优先级 |
|----------|----------------|----------------|------------|
| **文档 / 知识库** | 项目文档、发布说明、开发规范、FAQ 需要可版本化、可检索的承载；与项目/版本关联后便于新人上手与审计。 | 版本管理有 Changelog，但偏「单版本说明」；缺少按项目或全局的文档树、富文本/Markdown、权限与历史。 | **P1** |
| **里程碑 / 发布计划** | 按项目规划「版本计划、上线日、里程碑」，与任务、版本管理联动，支撑排期与复盘。 | 在版本管理、项目扩展方向中提及，但未单独成模块。独立模块可提供：里程碑列表、计划上线日、关联版本/任务、日历视图。 | **P1** |
| **问题 / 缺陷跟踪** | 任务偏「计划内工作」，问题/缺陷偏「异常、Bug、线上故障」；两者生命周期与字段不同，商业 PM 常分开管理。 | 当前仅有「任务」，无单独的 Issue 类型（如缺陷、需求、风险）。可扩展任务类型，或独立「问题」模块与项目/版本关联。 | **P2** |
| **交付物 / 验收** | 对外交付时需关联「合同/验收清单、交付物、版本」；甲方签字或验收状态留痕。 | 偏乙方向甲方交付场景；版本管理已有「版本」，可扩展「验收状态、交付清单」或单独轻量「交付管理」。 | **P2** |

### 15.3 数据与洞察维度

| 缺口模块 | 商业级为何常见 | 与现有模块关系 | 建议优先级 |
|----------|----------------|----------------|------------|
| **全局搜索 / 命令面板** | 跨模块快速找项目、任务、用户、版本、文档，提升效率；支持快捷键调起（如 Cmd+K）。 | 各模块自有列表与筛选，无统一入口。独立「全局搜索」或顶部搜索框+下拉结果，聚合各模块数据并做权限过滤。 | **P0** |
| **工作台 / 首页定制** | 首页不仅是入口，而是可配置的「我的工作台」：待办、最近项目、各模块摘要卡片、快捷操作。 | 现有首页以 APK 列表与统计为主；管理中心有面板。可升级为「可配置仪表盘」：拖拽组件、按角色/用户保存布局。 | **P1** |
| **审计与合规报表** | 除操作日志外，需要「谁在何时访问了哪些资源」的汇总报表、导出审计、敏感操作统计，满足合规与巡检。 | 操作日志已有明细；缺按「访问类型、资源类型、用户」聚合的报表与导出。可在操作日志中增加「报表/统计」子页，或独立「审计报表」模块。 | **P2** |

### 15.4 治理与运维维度

| 缺口模块 | 商业级为何常见 | 与现有模块关系 | 建议优先级 |
|----------|----------------|----------------|------------|
| **备份与恢复** | 数据备份策略、备份记录、恢复演练，确保故障可回滚；满足等保或客户对「可恢复」的要求。 | 当前依赖 data 目录与手工备份。独立模块可提供：定时备份配置、备份列表、一键恢复（或生成恢复脚本）、保留策略。 | **P1** |
| **集成与 API** | 开放 API、Webhook、与 Jira/GitLab/飞书等集成，便于与现有研发体系打通；商业平台常见「集成中心」。 | 当前若有集成多为写死在代码中。独立模块：API 文档、密钥管理、Webhook 配置、已配置集成列表与日志。 | **P1** |
| **许可证 / 授权** | 商业化销售时需控制授权数量、到期日、按 license 开启/关闭功能。 | 内部部署可不需要；对外 SaaS 或商业发行时常见。 | **P2** |

### 15.5 体验与开放维度

| 缺口模块 | 商业级为何常见 | 与现有模块关系 | 建议优先级 |
|----------|----------------|----------------|------------|
| **帮助中心 / 引导** | 帮助文档、新手引导、FAQ、操作说明，降低支持成本、提升上手速度。 | 当前无统一帮助入口。可独立「帮助中心」页（静态或简单 CMS），或仅在关键页提供「？」与链接。 | **P1** |
| **反馈与工单** | 用户提交问题/建议，管理员处理与回复，形成闭环。 | 无。轻量做法：反馈表单+写入数据表+简单列表与状态；或对接现有工单系统。 | **P2** |

### 15.6 缺口汇总与实施建议

- **建议优先补齐（P0）**  
  - **个人中心 / 个人设置**：与系统设置区分，提供修改密码、通知偏好、头像等统一入口。  
  - **全局搜索 / 命令面板**：跨模块检索项目、任务、用户、版本，并做权限过滤。

- **建议在中期规划（P1）**  
  组织架构、文档/知识库、里程碑/发布计划、工作台定制、备份与恢复、集成与 API、帮助中心。可根据客户或交付场景择优先做。

- **可选或按需（P2）**  
  角色模板、问题/缺陷跟踪、交付物/验收、审计合规报表、许可证、反馈与工单等；在明确需求或商业化阶段再纳入。

- **与现有模块的融合方式**  
  - 部分能力可**合并到现有模块**（如角色模板进用户管理、审计报表进操作日志、个人设置中的「通知偏好」进通知中心）。  
  - 若希望菜单与权限边界更清晰，或该能力会持续扩展，则**单独成模块**更利于商业级产品形态。
