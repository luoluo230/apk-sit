# Safe Change Workflow (Windows/Web-Jenkins-Unity)

## 0. 目标
- 新功能优先，但绝不破坏已有功能。
- 每次改动必须“可验证、可回退、可定位”。

## 1. 改动边界
- 单次只做一个功能点，不混改 UI 文案/布局/业务逻辑。
- 禁止在同一提交里顺手清理乱码、重构、改命名（除非用户明确要求）。
- 若发现历史脏数据或乱码：先记录，不在当前功能任务里处理。

## 2. 编辑策略
- 优先最小补丁（最少行数、最少文件）。
- 涉及字符串拼接的前端 JS：
  - 禁止复杂引号嵌套拼接。
  - 优先模板字符串或 DOM API（createElement）。
- 非 ASCII 文案改动需谨慎：若文件存在编码风险，优先用 ASCII 占位后再统一处理。

## 3. 强制检查（每次改完必做）
- Python 文件：`python -m py_compile <file>`
- 页面内嵌 JS：
  1. 从后端模板“渲染后”抽出 `<script>`
  2. `node --check rendered.js`
- 若任一步失败：不得重启服务，不得交付，先修到通过。
- 检查通过后、交付前：必须先提交再推送，禁止只 push 不 commit。
  - 必做：`git add` -> `git commit -m "<message>"` -> `git push origin <current-branch>`。
  - 若本次改动涉及多个仓库（如 `E:\maclient` 与 `E:\web\apk-site`），每个仓库都必须分别完成一次提交与推送。
  - 禁止把 `apk-site` 的改动只记在 `maclient` 仓库里，或反之。

## 4. 运行验证
- 重启服务后必须检查：
  - `GET /health` = ok
- 新功能必须提供“可见反馈”：
  - 按钮点击后至少出现 loading/success/fail 文案之一。
- 至少验证 1 条正向路径 + 1 条失败路径（例如接口失败时提示）。
- 任何功能修改完成后，必须执行一次服务重启再交付；禁止“未重启仅靠代码静态检查”即宣称完成。

## 5. 控制台报错处理规范
- 用户给出 console 报错后：
  - 先按行号定位（文件/行/列）
  - 只修该报错链路
  - 修复后再次跑 `node --check rendered.js`
- 禁止“未复现先大改”。

## 6. 回滚机制
- 改动前保留基线文件副本（同目录 `.bak` 或已知 clean 版本路径）。
- 若出现页面不可用：
  - 立即恢复基线
  - 再做增量补丁
- 禁止连续叠加修补超过 2 轮不清理；第 3 轮必须回基线重做。

## 7. 交付格式（必须包含）
- 改了哪些文件
- 做了哪些命令级校验（列命令和结果）
- 用户如何复测（3 步内）
- 若未完成：明确卡点和下一步

## 8. 禁止项
- 不验证就说“已修复”
- 只验证源码，不验证渲染后 JS
- 将多个需求混在一次补丁里
- 发现报错后继续叠加无关改动

## 9. 对外文案与本地化
- 面向用户可见内容必须为中文：按钮文字、状态提示、接口说明文案、错误提示、帮助文案。
- 若为临时调试文案，提交前必须恢复为中文正式文案。

## 10. 兼容性与影响面评估
- 做新功能时，必须评估已有调用方是否复用该接口/逻辑（页面、脚本、定时任务、其他模块）。
- 新增或修改接口参数时：
  - 默认向后兼容（旧参数可继续使用）。
  - 变更前给出影响清单与回归点。
- 禁止只修“当前页面可用”而忽略同接口其他入口。

## 11. 架构与可维护性约束
- 新功能禁止把大量逻辑堆入单一脚本/单一函数。
- 需要按职责拆分：路由层、服务层、前端交互层分别处理。
- 前端交互代码超过可读阈值时，优先拆为独立函数（初始化、事件绑定、请求、渲染）。
- 后端新增能力优先放在 service 层封装，路由仅做参数校验与编排。

## 12. 事故复盘硬规则（本项目新增）
- 用户明确要求“回退”时：只允许执行回退，不允许夹带任何试修、优化、重构。
- 回退动作前必须先锁定“唯一回退点”（构建号/时间点/截图节点），未锁定不得写文件。
- 连续 2 次修复失败后必须停手：输出当前状态、已改文件、下一步唯一动作，等待用户确认。
- 任何让用户重跑 Jenkins 之前，必须先本地通过对应语法检查：
  - Shell：`bash -n <script>`
  - Python：`python -m py_compile <file>`
- 禁止“边改边试”扩散：每轮只改一个链路、一个目标文件集合。
- 若当前改动引发新故障，优先恢复到上一可用状态，再进行下一步。
- 高压场景沟通规则：先确认目标再执行，不做争辩，不给不确定承诺。
- 交付时必须明确写出“做了什么、没做什么”，避免误解为已完成全量修复。
